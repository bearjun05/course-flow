# CourseFlow — 강의 제작 관리 대시보드

## 프로젝트 개요

온라인 강의 제작 프로세스(기획→교안→리허설→촬영→편집/검수→롤아웃)를 관리하는 PM 대시보드.
현재 프론트엔드만 존재하며, 백엔드는 나중에 별도 개발자가 연결 예정.

## 사용자

- 비개발 실무자 (PM/기획). 코드를 모르며 자연어로만 소통한다
- 한국어로 간결하게 응답. 기술 용어 없이 바로 작업
- "이 화면에서 이렇게 바꿔줘", "이 버튼 누르면 이렇게 되어야 해" 식으로 요청함

## 개발 명령어

모든 명령은 `frontend/` 디렉토리에서 실행:

```bash
cd frontend
npm run dev      # 개발 서버 (localhost:3000)
npm run build    # 프로덕션 빌드 (작업 완료 시 반드시 확인)
npm run lint     # ESLint
```

## 기술 스택

- Next.js 16 (App Router) + React 19 + TypeScript (strict)
- Tailwind CSS v4 + shadcn/ui (base-nova 스타일)
- 경로 alias: `@/*` → `./src/*`
- shadcn 추가: `cd frontend && npx shadcn@latest add <component-name>`

## 아키텍처

```
frontend/src/
├── app/                    # 페이지
│   ├── page.tsx            # 대시보드 (메인, 모든 상태 관리)
│   └── projects/
│       ├── [id]/page.tsx   # 강의 상세
│       └── new/page.tsx    # 제작 요청 폼
├── components/
│   ├── layout/             # 사이드바, 헤더
│   ├── dashboard/          # 칸반 보드, 프로젝트 그리드
│   │   ├── kanban/         # 4열 칸반 (교안작성/리허설/제작/롤아웃)
│   │   └── project-grid/   # 프로젝트 리스트 (flat/column/section 뷰)
│   ├── detail/             # 강의 상세 (일정/정보/영상피드백 탭)
│   ├── form/               # 제작 요청 폼
│   └── ui/                 # shadcn 기본 컴포넌트
└── lib/
    ├── types.ts            # 모든 타입 정의 ⚠️ 구조 변경 시 주의
    ├── constants.ts        # 칸반 컬럼, 상태값, D-Day 그룹, 색상 매핑
    ├── mock-data.ts        # 목 데이터 ⚠️ 백엔드 인터페이스 역할, 구조 유지
    └── process-helpers.ts  # 공정 관련 유틸
```

## 도메인 용어

| 용어 | 의미 |
|------|------|
| 프로젝트 | 하나의 강의 제작 단위 |
| 챕터 | 강의 내 섹션. 챕터별로 촬영→편집→자막→검수 태스크 반복 |
| 강(講) | 챕터 내 최소 단위 (1-1 = 1장 1강). 피드백/검수의 기본 단위 |
| 롤아웃 | 강의 출시(배포) |
| 신호등 (TrafficLight) | 프로젝트 건강도 (green/yellow/red) |
| 사업부 (BusinessUnit) | KDT / KDC / 신사업 |
| 공정 (TaskType) | 교안제작→리허설→촬영→편집→자막→검수→업로드→롤아웃 |
| 커기매 | 커리큘럼기획 매니저 (약어) |

## ⚠️ 핵심 규칙 1: 정책 자동 문서화

> 이 프로젝트에서 **가장 중요한 규칙**이다. 나중에 백엔드 개발자가 이 프론트를 받아서 API를 연결할 때, `docs/policies/`가 유일한 비즈니스 로직 참고 자료가 된다.

### 감지 대상

사용자 발화에서 다음 패턴을 감지하면 반드시 `docs/policies/`에 문서화:

| 패턴 | 예시 |
|------|------|
| 조건부 규칙 | "촬영 완료되면 편집으로 넘어가야 해" |
| 절대 규칙 | "검수 없이 롤아웃하면 절대 안 돼" |
| 상태 전이 | "리뷰에서 반려되면 다시 편집으로 돌아가" |
| 권한/역할 | "편집자는 검수 상태를 변경할 수 없어" |
| 계산 로직 | "마감 3일 전부터 노란불" |
| 분류 기준 | "KDT는 챕터당 30분 이상이면 분할" |
| 데이터 관계 | "한 강에 영상이 여러 개일 수 있어" |
| 업무 흐름 | "편집자 구인은 리허설 이후에 해" |

**단순 UI 요청("색상 바꿔줘", "여백 넓혀줘")은 정책이 아니다. 기록하지 않는다.**

### 기록 방법

1. `docs/policies/`에서 해당 카테고리 파일을 찾거나 새로 생성
2. 아래 형식으로 규칙 추가:

```markdown
## 규칙
- **[P-001]** 규칙 내용
  - 배경: 사용자가 설명한 이유 (있다면)
  - 적용 화면: 이 규칙이 반영되는 UI 위치

## 예외 사항
- 예외 케이스

## 미결정 사항
- [ ] 아직 확인 필요 (→ 사용자에게 질문할 것)

## 관련 화면
- 해당 페이지/컴포넌트
```

3. `docs/policies/README.md` 인덱스 업데이트
4. **코드에도 즉시 반영** (UI 동작에 규칙 적용)

### 정책 ID 규칙

카테고리별 접두사 사용:
- `ST-` 상태 전이 (State Transition)
- `RL-` 역할/권한 (Role)
- `PR-` 프로세스 (Process)
- `RV-` 검수/품질 (Review)
- `SC-` 일정/계산 (Schedule/Calc)
- `DT-` 데이터 규칙 (Data)
- `BZ-` 기타 비즈니스 규칙 (Business)

## ⚠️ 핵심 규칙 2: 정책 갭 감지 및 질문

UI를 구현할 때 비즈니스 규칙이 불명확하면 **반드시** 사용자에게 자연스럽게 질문한다.

### 질문해야 하는 상황

- **상태 전이 조건 미정의**: "촬영 완료로 바꾸려면 어떤 조건이 필요한가요?"
- **예외 케이스 미정의**: "마감일이 지난 프로젝트는 어떻게 표시하나요?"
- **권한 불분명**: "이 기능은 PM만 쓰나요, 편집자도 쓰나요?"
- **데이터 관계 모호**: "한 프로젝트에 튜터가 여러 명일 수 있나요?"
- **우선순위 미정의**: "칸반에서 카드 순서는 뭘 기준으로 정렬하나요?"
- **경계값 미정의**: "노란불 기준이 마감 며칠 전인가요?"

### 질문 방식

- 작업 흐름 중에 **자연스럽게** 끼워서 질문 (별도 세션으로 몰아서 하지 않음)
- 한 번에 **최대 2개**까지만 질문
- 답변을 받으면 즉시 `docs/policies/`에 기록 + 코드 반영

## 작업 규칙

### 작업 완료 시 자동 커밋

의미 있는 작업 단위가 끝나면 반드시:
1. `cd frontend && npm run build` 빌드 확인
2. 변경된 파일 **모두** (코드 + 정책 문서) git add
3. 한국어 커밋 메시지로 커밋 (예: "칸반 보드에 드래그앤드롭 추가")

### 기획 문서 참조

`docs/` 디렉토리에 기획 문서가 있다. 새로운 기능 작업 전에 관련 문서를 반드시 확인:
- `docs/README.md` — 전체 기획서 (데이터 모델, 역할, 프로세스)
- `docs/상태값목록.md` — 프로젝트/태스크 상태값 정의
- `docs/PM-워크플로우.md` — PM 업무 흐름 (Phase 0~6)
- `docs/영상-피드백-상세-설계.md` — 영상 검토 체크리스트 설계
- `docs/policies/` — 축적된 비즈니스 정책 (가장 최신)

### UI 작업 원칙

- 새 컴포넌트는 기존 패턴 따르기 (shadcn/ui + Tailwind)
- `lib/types.ts` 타입 추가는 OK, 기존 타입 구조 변경은 신중하게
- `lib/mock-data.ts` 구조는 가급적 유지 (나중에 백엔드 API 인터페이스 역할)
- 새 shadcn 컴포넌트 필요 시: `cd frontend && npx shadcn@latest add <name>`

### 확인 및 배포

**1. 로컬 확인 (작업 중 빠르게 확인)**
```bash
cd frontend && npm run dev
# 브라우저에서 http://localhost:3000 접속
```
- 코드 수정하면 자동 반영됨. 작업하면서 바로바로 확인할 때 사용

**2. 배포 (다른 사람에게 공유)**
```bash
git add -A && git commit -m "변경 내용" && git push
```
- push하면 https://course-flow-lake.vercel.app/ 에 자동 배포 (1~2분 소요)
- 유관계자에게 이 링크를 공유하면 별도 설치 없이 바로 확인 가능
