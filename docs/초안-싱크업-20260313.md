## 1. 프로젝트 배경 — 왜 만드는가

### 문제 정의

| # | 문제 | 현재 상황 |
|---|------|----------|
| 1 | **제작 현황을 확인할 수 있는 시스템이 없다** | 유관계자(커기매, 운영기획매니저, 튜터, 편집자/검수자)가 제작 현황을 알려면 PM에게 직접 물어봐야 한다 |
| 2 | **PM이 모든 현황을 수동으로 관리하고 있다** | PM이 하루 1회 노션 업데이트 + 슬랙 공유. 강의 수 증가 시 PM 리소스가 비례 증가하여 온타임 딜리버리 리스크가 커진다 |

### AS-IS → TO-BE

```
AS-IS (시스템 없음)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  튜터/편집자               PM                유관계자
  (작업 완료)          (하루 1회 수동)       (현황 파악)

    작업 완료  ─────→  노션 업데이트  ─────→  "지금 어디까지 왔어요?"
                       슬랙 공유               ← PM에게 직접 문의
                       주 7시간 소요            1~3일 지연
```

```
TO-BE (시스템 도입)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  튜터/편집자              시스템              유관계자
  (작업 완료)          (실시간 반영)         (현황 파악)

    상태 직접 변경 ──→  대시보드 자동 반영 ──→  시스템에서 직접 확인
                       PM은 일일 검증만          PM 문의 불필요
                       주 2시간 이하              실시간 확인 가능
```

| | AS-IS | TO-BE |
|---|---|---|
| **현황 확인 방법** | PM에게 슬랙으로 문의 | 시스템 대시보드에서 직접 확인 |
| **상태 업데이트 주체** | PM 혼자 (하루 1회) | 튜터/편집자가 직접 변경 |
| **상태 반영 속도** | ~24시간 | 수 시간 이내 (이벤트 기반) |
| **PM 현황 관리 시간** | 주 7시간 | 주 2시간 이하 (검증만) |
| **정보 접근성** | PM을 통해서만 가능 | 시스템 접속 시 누구나 즉시 |

## 2. 이번 MVP 목표

> **제작 현황을 시스템에서 실시간으로 확인할 수 있는 상태를 만든다**

이 한 문장이 이번 MVP의 전부입니다.

- 유관계자가 시스템에 접속하면 **현재 강의 제작 현황이 보인다**
- 튜터가 직접 상태를 변경하여 **PM 의존 없이 실시간 반영된다**
- PM은 수동 업데이트 대신 **일일 검증만 수행한다**

---

## 3. KPI

### 핵심 KPI — 상태 업데이트 지연시간

> 작업 상태가 실제로 변경된 시점 → 시스템에 반영되는 시점까지의 평균 시간

| | AS-IS | TO-BE 목표 |
|---|---|---|
| **상태 반영 지연시간** | **~24시간** (PM 하루 1회 수동 업데이트) | **실시간** (작업자 직접 변경) |

- AS-IS: 튜터가 촬영 완료 → PM이 인지 → PM이 다음 날 노션 업데이트 → ~24시간 지연
- TO-BE: 튜터가 촬영 완료 → 시스템에서 직접 상태 변경 → 즉시 반영
- 측정: 시스템 상태 변경 로그 시각 기준으로 자동 측정 가능

이것이 핵심 KPI인 이유: **상태가 실시간으로 반영된다 = 시스템에서 실시간으로 확인할 수 있다.** MVP 목표 그 자체를 직접 측정하는 지표입니다.

### 선행 지표 — 시스템 정보 정확도

핵심 KPI(실시간 반영)가 제대로 작동하는지 검증하는 지표:

| 선행 지표 | 정의 | 목표 |
|---|---|---|
| **일일 정보 정확도** | PM이 매일 1회 검증 시, 시스템 상태와 실제 상태의 일치율 | **90% 이상** |

- 측정: PM 일일 체크 시 `일치 건수 / 전체 관리 프로젝트 수 × 100`
- 실시간으로 반영되더라도 정확하지 않으면 의미 없음 → 정확도가 선행 조건

### 후행 지표 — PM 리소스 절감

핵심 KPI가 달성되면 자연히 따라오는 결과:

| 후행 지표 | 정의 | 목표 |
|---|---|---|
| **PM 현황 관리 소요시간** | 상태 업데이트 + 공유에 쓰는 주간 시간 | **주 7시간 → 2시간 이하** |

### KPI 간 관계

```
작업자(튜터/편집자)가 직접 상태 변경 (구조 변화)
        │
        ▼
상태 업데이트 지연시간: 24시간 → 실시간 (핵심 KPI)
        │
        ├──→ 일일 정보 정확도 90%+ (선행 지표: 빠를 뿐 아니라 정확한가?)
        │
        ├──→ 유관계자가 시스템에서 실시간 확인 가능 (MVP 목표 달성)
        └──→ PM 현황 관리 시간 주 2시간 이하 (후행 지표)
```
